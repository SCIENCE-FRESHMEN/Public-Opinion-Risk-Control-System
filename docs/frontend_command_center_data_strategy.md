# 首页与单股详情页数据替换策略

## 1. 目标
本策略用于指导成员 D 对以下两个答辩核心页面进行“真实数据优先、静态数据兜底”的改造：

- `frontend/src/pages/command-center-page.tsx`
- `frontend/src/pages/stock-insight-page.tsx`

目标不是一次性把所有静态数据全部删掉，而是先完成第一轮高价值替换：

- 核心指标走真实接口
- 核心图表走真实接口
- 目前后端尚未直接提供的数据继续保留 fallback

这样既能形成“前端已接真实数据”的证据，又能保证页面在答辩和截图场景下稳定不空白。

## 2. 策略原则

### 2.1 真实数据优先
只要现有后端接口已经提供了可直接复用的数据，就优先接入，不继续依赖 `real-stock-data.ts`。

### 2.2 fallback 保底
如果现阶段后端还没有直接提供页面所需字段，则继续使用 `frontend/src/features/command-center/real-stock-data.ts` 兜底，保证页面稳定展示。

### 2.3 页面层不直接拼乱接口
页面组件不应散乱地在多个位置直接处理接口差异，而应通过轻量聚合层或 adapter，把真实接口与 fallback 统一映射成页面需要的数据结构。

### 2.4 先高价值，后全面化
第一轮只做最值钱的页面和最能体现“真实联调”的模块，不追求一次性全量重构。

## 3. 当前页面状态判断

### 3.1 `command-center-page.tsx`
当前首页视觉完成度高，但核心数据默认来自：

- `realCommandCenterDashboard`
- `realStockInsight`

这说明首页当前本质上是“高保真静态壳”。

### 3.2 `stock-insight-page.tsx`
当前单股详情页结构已经成熟，但默认数据来自：

- `realStockInsight`

这说明详情页也是“高保真静态壳”。

## 4. 首页替换策略

## 4.1 第一层：立即切真实接口
这些内容优先改成真实接口驱动：

- 顶部时间
- 当前监测标的名称
- 当前价格
- 当前价格涨跌幅
- 当前情绪得分
- 当前风险等级

建议主要复用：

- `/api/overview`
- `/api/linkage`
- `/api/meta/filters`

### 对应目标
让首页第一眼看上去就已经不是纯静态页面，而是“实时从现有数据产物计算出的结果页”。

## 4.2 第二层：半真实拼装
这些内容优先通过已有接口拼出一版真实趋势：

- 焦点股联动图
- 价格趋势
- 情绪趋势
- 热度趋势

建议复用：

- `/api/linkage`

但可继续保留：

- 主题标签先用 `real-stock-data.ts`

### 对应目标
让首页最关键的联动图具备真实数据来源，同时不因为主题字段暂时没统一而阻塞推进。

## 4.3 第三层：暂时保留 fallback
这些内容当前先不强行真实化：

- 舆情热度榜 Top 10
- 风险事件列表
- 谣言事件数
- AI 研判简报卡
- 数据链路说明文案中的部分统计文字

当前先继续使用：

- `real-stock-data.ts`

原因：

- 后端还没有为首页直接提供完整聚合接口
- A / B / C 的结果文件还未完全收敛
- 这些模块对答辩重要，但对“真实联调证据”的优先级低于核心指标和联动图

## 5. 单股详情页替换策略

## 5.1 第一层：立即切真实接口
这些内容优先改成真实接口驱动：

- 最新价格
- 价格涨跌幅
- 情绪得分
- 情绪标签
- 联动趋势图

建议主要复用：

- `/api/overview`
- `/api/linkage`

### 对应目标
让详情页的“核心分析区”真正由后端数据产物驱动。

## 5.2 第二层：半真实拼装
这些内容优先接新闻相关真实数据：

- 代表性舆情摘要
- 新闻时间
- 来源
- 情绪标签

建议复用：

- `/api/news/drilldown`

### 对应目标
让详情页能证明“页面不是凭空写文案，而是已经从真实新闻样本中取到了内容”。

## 5.3 第三层：暂时保留 fallback
以下模块当前继续保留：

- 风险时间线
- 谣言占比
- 主题词权重
- AI 个股简报
- 部分项目证据链说明数据

继续使用：

- `real-stock-data.ts`

原因：

- 这些字段需要 A/B/C 的结果继续补齐
- 当前后端没有直接提供一整套单股聚合接口
- 保留 fallback 可以保证详情页在答辩时完整、稳定、可截图

## 6. 建议的数据聚合层

## 6.1 首页聚合层
建议新增一个首页聚合 hook 或 adapter，负责把：

- `/api/overview`
- `/api/linkage`
- 当前 filters 信息
- `real-stock-data.ts` fallback 数据

统一映射成首页需要的数据结构。

### 作用
- 页面组件不直接感知多个接口差异
- 缺字段时自动 fallback
- 后续 A/B/C 数据补齐后，只改 adapter，不改页面结构

## 6.2 单股详情页聚合层
建议新增一个详情页聚合 hook 或 adapter，负责把：

- `/api/overview`
- `/api/linkage`
- `/api/news/drilldown`
- `real-stock-data.ts` fallback 数据

统一映射成详情页需要的数据结构。

### 作用
- 先完成“部分真实、部分兜底”的平滑替换
- 后续真实字段补齐时，不需要大规模重写详情页 JSX

## 7. 第一轮改造实施范围
本轮只做以下内容：

1. 为首页建立真实数据 + fallback 的聚合策略
2. 为单股详情页建立真实数据 + fallback 的聚合策略
3. 优先接核心指标与联动图
4. 暂不强行替换热度榜、AI 简报卡、谣言占比等高依赖模块

本轮不做：

- 全量删除 `real-stock-data.ts`
- 新增复杂后端接口
- 一次性重构所有页面

## 8. 对 A / B / C 的依赖落点

### 对 B 的依赖
后续将影响：

### 对 C 的依赖

## 9. 当前实现技术栈口径
当前前端实现统一描述为：

- `React + Vite + Recharts + FastAPI`

在报告、PPT、讲稿中，禁止把当前实现写成：

- Vue
- Element Plus
- ECharts

## 10. 结论
本策略选择“折中方案”：

- 核心指标和图表走真实接口
- AI 简报、谣言占比、部分事件卡先保留 fallback

这是当前最适合成员 D 的推进路径，因为它同时满足：

- 有真实联调证据
- 页面不空白
- 截图稳定
- 后续可平滑继续替换

下一步就进入真正的前端代码改造阶段。
